## 1. システム概要

チラシのポスティング実績を、地図上のピンで記録・管理する Web アプリ。
「配布場所（location）」と「配布履歴（visit）」を分離したリレーショナル設計とし、
各場所のピンの色は「その場所の直近の配布評価」で出し分ける（大島てる的な体験）。

- 想定利用者: 少人数のチーム（相互に閲覧・編集し合う）
- 端末: スマートフォン主体（PC でも利用可、PC は一覧をテーブル表示）

```mermaid

  locations  ||--o{ visits      : "1場所に複数の配布履歴"
  visits     ||--o{ visit_logs  : "変更を監査記録"
  profiles   ||--o{ visits      : "配布者"
  profiles   ||--o{ visit_logs  : "操作者"

  schools {
    uuid id PK
    text name
    float lat
    float lng
  }
  locations {
    uuid id PK
    uuid school_id FK
    text spot "立ち位置"
    float lat
    float lng
  }
  visits {
    uuid id PK
    uuid location_id FK
    uuid user_id FK
    date date
    time start_time
    time end_time
    int count "配布枚数"
    text rating "good/normal/bad"
    bool is_deleted "論理削除"
    timestamptz created_at
    timestamptz updated_at
  }
  visit_logs {
    uuid id PK
    uuid visit_id FK
    uuid user_id FK
    text action "create/update/delete"
    jsonb before_data
    jsonb after_data
  }
  profiles {
    uuid id PK
    text email
    text display_name
  }
  map_notes {
    uuid id PK
    float lat
    float lng
    text color
    text label
    text memo
  }
```

- `map_notes` は配布実績とは独立した、地図上の汎用メモ（色＋見出し＋メモ。色に固定の意味は持たせない）。
- テーブルの列定義詳細は [REQUIREMENTS.md](../REQUIREMENTS.md) 4章、実体は [supabase/schema.sql](../supabase/schema.sql)。

### 主要ロジック
- **ピン色決定**: ビュー `location_pin_status` が、各 location の未削除 visits のうち
  「date, start_time が最新」の1件の rating を返す。緑=good / 灰=normal / 赤=bad / 履歴なし=グレー。
- **監査ログ**: `visits` への INSERT/UPDATE をトリガー `log_visit_changes` が捕捉し、
  `visit_logs` に create/update/delete と before/after(jsonb) を自動記録。
- **論理削除**: 物理削除は行わず `is_deleted` フラグで管理。RLS でも DELETE を許可しない。

### デモ表示モード（スキーマ分離）

本番と同じ Supabase プロジェクト内に、同じ構造の **`demo` スキーマ**を用意している。
ログイン画面のチェックボックスで、アプリの参照先を `public`（本番）/ `demo`（架空データ）に切り替える。

```mermaid
flowchart LR
  CB["ログイン画面<br/>「デモデータを参照する」"] -->|"cookie: demo_mode"| CC["createClient<br/>唯一の切替点"]
  Q["queries.ts / actions.ts<br/>変更不要"] --> CC
  CC -->|"OFF"| PUB[("public スキーマ<br/>本番データ")]
  CC -->|"ON"| DEM[("demo スキーマ<br/>架空データ")]
```

**なぜフラグ列（`is_demo`）ではなくスキーマ分離か**

参照の切替はチェックボックス（＝クライアント由来）で行うため、**RLS では制御できない**
（RLS は JWT＝「誰か」でしか出し分けられず、画面のチェックボックスを条件にできない）。
したがって分離の境界はアプリ側になるが、その前提では:

| 方式 | 絞り忘れたときに起きること |
|---|---|
| 同一テーブル＋`is_demo` フラグ | **本番データがデモ画面に出る**（同居しているため） |
| **スキーマ分離（採用）** | demo を参照している限り**本番の行は物理的に出てこない** |

加えて、切替点が `createClient()` の**1か所に閉じる**ため（`supabase-js` の `db.schema` 指定）、
`queries.ts` / `actions.ts` は**無変更**で済む。`.from("visits")` がそのまま `demo.visits` を指す。

**設計上の割り切り**
- チェックボックスは**セキュリティ境界ではなく利便スイッチ**（デモは自分の端末で見せる前提）。
  本番データを守っているのは「demo スキーマに本番の行が存在しないこと」。
- デモ中は常時バナーを表示するが、**ワンタップの解除ボタンは意図的に置かない**。
  人に見せている画面に「押すと本番データが出るボタン」があると誤タップで露見するため。
  解除はログアウト → ログイン画面でチェックを外す、という意図的な手順を踏ませる。
- `demo` スキーマでは `auth.users` への外部キーを張らない（架空メンバーを seed するため）。
  デモ内の整合性（visits→locations 等）は維持する。
- **`public` にテーブル・カラムを追加したら `demo` にも同じ DDL を当てる**（スキーマのドリフト防止）。

実体は [supabase/schema_demo.sql](../supabase/schema_demo.sql)（構造）と
[supabase/seed_demo.sql](../supabase/seed_demo.sql)（架空データ）。
デモの座標は本番の活動エリアから離した場所に置いている（地図アプリでは名前より場所の方が機微なため）。

## 5. 認証・認可設計

**通常ログインはメール＋パスワード**（`signInWithPassword`）。メールリンク（マジックリンク）は
初回ログイン・パスワード未設定者向けのフォールバックとして残している。ログイン後、ヘッダの
パスワード設定ダイアログ（`updateUser`）で自分のパスワードを設定できる。

### メールリンク認証シーケンス（初回ログイン時）

```mermaid
sequenceDiagram
  participant U as ユーザー
  participant App as アプリ（Next.js）
  participant Sb as Supabase Auth

  U->>App: メールアドレスを入力・送信
  App->>Sb: signInWithOtp(shouldCreateUser=false)
  alt 登録済メンバー
    Sb-->>U: ログインリンクをメール送信
    U->>App: リンクを開く（/auth/confirm）
    App->>Sb: セッション確立（code / token_hash 検証）
    Sb-->>App: セッション（Cookie）
    App-->>U: 一覧へ遷移
  else 未登録メール
    Sb-->>App: エラー（作成不可）
    App-->>U: 「登録されていません」案内
  end
```

### 認可（アクセス制御）
- **Middleware**: 全リクエストでセッションを検証。未認証は `/login` へリダイレクト。
- **RLS（Row Level Security）**: 全テーブルで有効。`authenticated` ロールのみ参照・更新可、
  `anon`（未認証）は一切アクセス不可。`visits` は物理 DELETE 不可（論理削除運用）。
- **サインアップ制御**: Supabase 側で新規サインアップを無効化。メンバーは管理者が Users から追加。
- **監査**: 誰でも編集・削除できるが、操作は必ず `visit_logs` に記録される。

## 7. 非機能・運用

- **セキュリティ**: RLS による行レベル制御、パラメータ化クエリ（SQLインジェクション対策）、
  React による自動エスケープ（XSS対策）、公開キー前提でのサインアップ無効化。
- **デプロイ構成**: Vercel（本番 `flyer.〜`）＋ Supabase 1プロジェクト。
  デモはアプリ内のスキーマ切替で実現するため、**別プロジェクト・別デプロイ・専用URLは不要**。
- **スキーマ管理**: `supabase/schema.sql`（public）/ `supabase/schema_demo.sql`（demo）はいずれも
  冪等（再実行可能）に記述し、常に全実行で最新化できる。破壊的変更の前は
  `begin; … rollback;` で予行演習し、戻しSQLを用意する（各SQLファイル末尾に記載）。
- **バックアップ**: Supabase Free プランには日次バックアップが無い。本番データを運用するため、
  定期的な手動エクスポート（`pg_dump` 等）を行うか、Pro プランへの移行を検討する。
- **品質管理**: [テスト仕様書](./テスト仕様書.md) / [変更管理表](./変更管理表.md) / [バグ管理表](./バグ管理表.md)
  / [コーディング規約](./コーディング規約.md)。
