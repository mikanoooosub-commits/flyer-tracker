# 構想・ロードマップ（チラシ配布実績管理アプリ）

現行アプリ（配布実績の地図記録・共有）をベースにした、将来の拡張構想メモ。
※ 実装前のアイデア段階。優先度・実現方法は今後精査する。

## 位置づけ
- 直近は **LP（ランディングページ）を優先**し、現行アプリはまず**実運用で使ってみる**フェーズ。
- 以下は「使ってみて効果が見えたら着手する」候補。

---

## 構想一覧

### 1. 一斉下校ガイド（配布の“狙い目”提案）
- **目的**: できるだけ多くの学年が**一斉に下校する日・学校**を狙って配布に行きたい。その日/学校をガイドする。
- **背景データ**: 各小学校が HP で公開する**月間行事予定PDF**（板橋区は毎月更新）。表形式で「日・曜・行事・**学年別の授業時間数（コマ数）**（1〜6年＋特別支援学級）」が載っている。コマ数が少ない＝早帰り、多くの学年が同時に少ないコマ数＝**一斉下校**＝配布好機。
- **実装の考え方**:
  - PDFから表を抽出 → `school_schedules(school_id, date, grade, periods)` のように学年×日付のコマ数を保持。
  - コマ数→実際の下校時刻は各校の**時程表**（1コマ何分・何時始まり）とセットで換算。
  - 「今日/近日、多くの学年が同時刻に下校する学校ランキング」を提示。
- **前提（判明した好条件）**:
  - **板橋区内は予定表のフォーマットがほぼ統一** → 1つのパーサーで区内全校をカバーできる見込み（当初の「学校ごとにバラバラ」懸念は板橋区内では小さい）。
  - **5時間授業のときの下校時刻もおおむね一定** → 各校の時程表が無くても、コマ数から下校時刻を近似できる。
  - 現状は**目視でチェック**している。アプリ化の価値は「多数の学校を横断して“今日の狙い目校”を自動ランキング」する点（人力では数校が限界）。
- **留意点 / 難所**:
  - **他区（北区・豊島区など）へ展開するとフォーマット差**に直面する可能性 → 区ごとにパーサー/取り込み方式を用意する構えが必要。
  - PDFがスキャン画像の場合はOCRが要る。まずは板橋の標準フォーマット（テキストPDF想定）で表抽出（pdfplumber / tabula 等）＋人手チェックの半自動から。

### 2. 配布タイミング最適化（門前配布の“クールダウン”）
- **目的**: 同じ場所での配布が短期間に続くと受け取り率が下がるため、適切な間隔を空けたい。「今日行くならココ」「ここはしばらく間隔を空ける」を提案。
- **背景データ**: **既存の配布履歴**（`visits`: 場所・日付・評価）だけで実現可能。
- **実装の考え方**:
  - 各 location の「最終配布からの経過日数」「配布頻度」「直近評価」からおすすめ度スコアを算出。
  - 地図・一覧に「おすすめ」「クールダウン中（例: 前回からN日未満）」をバッジ/色で表示。
- **留意点**: しきい値（何日空ければOK 等）は経験則なので**設定可能に**。既存データで動くため**着手ハードルが最も低い**。

### 3. 人口ヒートマップ（e-Stat）
- **目的**: 地図上で**ターゲット（未就学児・小学生）が多く住む地域**を可視化。**スマホから**、e-Stat の jSTAT MAP のような**町丁目単位のコロプレス**（例: 2020国勢調査 小地域 年齢5歳階級 0〜4歳）を見られるイメージ。
- **背景データ**: e-Stat（統計GIS）の**小地域（町丁字）× 年齢5歳階級人口**（国勢調査）。境界データ（Shapefile/GeoJSON）＋小地域集計CSVを **KEY_CODE（町丁字コード）で結合**。
- **実装の考え方**:
  - e-Statから対象区の境界データ＋人口CSVを取得 → GeoJSONに結合・簡略化（mapshaper/ogr2ogr）→ **静的アセット**として配信。
  - Leaflet に町丁ポリゴンをコロプレス表示（0〜4歳・5〜9歳などで色分け）。凡例つき。
- **留意点 / 好条件**:
  - **区スコープなら町丁ポリゴンは数百枚程度**で描画負荷は許容範囲（全国だと重いが、板橋区だけなら問題になりにくい）。
  - 更新は国勢調査5年ごと＝ほぼ静的。データ準備は一度やれば長く使える。
  - この**町丁境界データは構想4・5でも共通利用できる**（下記シナジー参照）。

### 4. 学区（校区）表示
- **目的**: 地図上で各小学校の**学区**を表示し、配布エリアの計画に使う。
- **背景データ**: 学区ポリゴン。**国土数値情報「学校区」データ（A27）** や自治体公開データが候補。
- **実装の考え方**: 学区GeoJSONを地図にオーバーレイし、学校マスタと紐付け。
- **留意点**: 学区データの入手性・精度・更新。区によって公開状況が異なる。

### 5. ポスティング記録への拡張
- **目的**: 小学生への直接配布だけでなく、**ポスティング**も記録。
- **記録の粒度（確定方針）**: **丁目単位**でよい。1レコード＝「日付・丁目・**戸建て何部**・**代表的なマンション何部**（＋マンション名/メモ）」。
  - 運用ルール: 「この丁目に配る」と決めたら**1番地から順に配る**ので、番地レベルの精緻な記録は不要。**GPSトラッキングも不要**。
- **実装の考え方**:
  - `postings(id, area_code/丁目, date, houses_count, mansion_count, memo, ...)` を追加（または記録種別で `visits` を拡張）。
  - 地図表示は**丁目の中心 or ポリゴン**で（構想3の町丁境界データを流用すれば個別ジオコーディング不要）。
  - 集計: 丁目別の配布部数、前回配布からの経過（重複配布の把握）。
- **留意点**: 番地までのジオコーディングは不要になったので**難易度が下がった**。丁目コードの持ち方を町丁境界データと揃えるのがポイント。

### 6. 配布セッション＋在庫連動（残部逆算方式）※あったらいいなメモ
- **目的**: 「持って行った部数」と「残って持ち帰った部数」から**配った枚数を逆算**し、あわせて**教室の在庫**を自動で減らす。どれだけ在庫が残っているか＝**新しくチラシセットを作るべきか**が一目で分かるようにする。
- **現場の運用フロー**:
  1. **配布現地に着いたら**、場所・日時を記録して「**開始**」ボタンを押す（開始時刻を自動記録）。あわせて**持って行ったチラシ**を記録：**男の子用 X部 / 女の子用 Y部**。
  2. 配り終わったら（**その場でも、事務所に戻ってからでも**）、**終了時刻・メモ・残った部数（男女それぞれ）**を記録。
  3. **配った枚数 = 持参 − 残部**（男女別に自動計算）。
  4. **教室の在庫 = 在庫 − 配った枚数** で自動減算 → 残在庫と「補充要否」が見える。
- **ポイント（なぜ良いか）**:
  - 配布数を直接入力させず**残部から逆算**するので、現場での入力が**ラクで正確**（数え間違い・入力忘れが減る）。
  - **男女別（性別ごとのチラシ）**という新しい軸が入る。
  - 「開始」ボタン方式で**開始時刻を自動記録**（現状の手入力を置き換え／補完）。
  - 在庫は**教室にある1つのプール**として管理（学校別ではなく全体在庫）。補充（チラシセット作成）で加算。
- **実装の考え方（着手時）**:
  - `visits` を拡張 or 「配布セッション」概念を新設：`brought_boys / brought_girls`（持参）・`left_boys / left_girls`（残部）を保持し、配布数は**導出**（`brought − left`）。
  - `inventory`（種別ごとの現在庫数）テーブルを追加し、セッション確定時に**配布数を減算**、補充時に**加算**。残在庫が閾値以下で「チラシセットを作る」アラート。
  - 既存の `count`（配布枚数）との整合：セッション方式では `count = 男 + 女` を自動算出して埋める、等の移行方針を決める。
- **留意点**: まずは**メモのみ（未実装）**。性別区分・在庫という業務ルールが増えるので、UI/スキーマは小さく試作してから本採用する。

## シナジー：町丁境界データが“共通の土台”
構想3（人口）・4（学区）・5（ポスティング）は、いずれも **町丁（小地域）ポリゴン**の上に成り立つ。
一度 e-Stat/国土数値情報から**町丁境界GeoJSON**を整備すれば、
- ③ 各町丁に人口を色付け
- ④ 学区ポリゴンを重ねる
- ⑤ ポスティングを町丁に集計・表示

…が同じ地図基盤で回る。**「町丁境界データの整備」を1つの共通タスク**として先にやると、後続がまとめて楽になる。

## データモデル拡張の方向（メモ）
- `school_schedules(school_id, date, grade, periods)` … 学年別コマ数（構想1）
- `school_bell_schedules` … 時程（コマ→時刻換算, 構想1）
- `postings` … ポスティング記録（構想5）
- 人口・学区は**外部GeoJSON/統計データのオーバーレイ**として扱う（DB化 or 静的配信を検討, 構想3・4）
- 現行の RLS・ID紐付け設計は、これらの追加テーブルにも素直に拡張できる想定。

## 着手順の私見（LP後・判明した好条件を反映）
1. **構想2（配布クールダウン）** — 既存データだけで動く・すぐ価値。最優先。
2. **構想1（一斉下校ガイド）** — 板橋区は同一フォーマット＆下校時刻ほぼ一定と判明し**実現性が上がった**。板橋限定でまず作り、多校横断ランキングで“今の目視”を自動化。他区展開時にフォーマット差を追加対応。
3. **町丁境界データ整備（共通タスク）** — ③④⑤の土台。ここを先に片付ける。
4. **構想5（ポスティング記録）** — 丁目単位＆GPS不要で軽くなった。町丁データが整えばすぐ。
5. **構想3（人口ヒートマップ）** — 区スコープなら描画・データ量とも現実的。
6. **構想4（学区表示）** — 学区GeoJSONが入手できれば重ねるだけ。
