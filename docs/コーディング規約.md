# コーディング規約（チラシ配布実績管理アプリ）

本書は本プロジェクトで守るコーディング上のルールをまとめたもの。
「なぜそうするか」を添えているので、迷ったら理由に立ち返って判断する。
設計は [設計書](./設計書.md)、今後の構想は [構想・ロードマップ](./構想・ロードマップ.md) を参照。

---

## 1. データベース / クエリ

### 1-1. `select *` は使わず、必要な項目だけ取る
- **理由**: 後からカラムを追加しても、取得側が意図せず影響を受けない。転送量も減り、型（TypeScript）と実際に使う項目が一致して読みやすい。
- **やり方**: Supabase では `.select("id, date, count, rating")` のように**列を明示**する。
- **例外**: 使い捨ての調査クエリや、明確に全列が欲しい管理用途のみ。本番コードには残さない。

### 1-2. カラム追加は「NULL許容」か「デフォルト値」のどちらかにする
- **理由**: 既存行にはそのカラムの値が無い。`NOT NULL` かつデフォルト無しで追加すると、
  その列を指定しない**既存コードの INSERT がエラーになる**（＝追加しただけで書き込みが壊れる）。
- **ルール**:
  - 追加カラムは `null` 許容にする、**または** `not null default <値>`（例 `default 0`）を付ける。
  - `schema.sql` では `alter table ... add column if not exists ...` の**追記型**で書く（冪等を維持）。

### 1-3. テーブル・カラムの削除／リネーム／型変更は原則しない（append-only）
- **理由**: 削除・リネームは古いコードを壊し、ロールバックが難しくなる。追加だけなら後方互換が保てる。
- **どうしても必要なとき**: 「新カラム追加 → コード移行 → 旧カラム廃止」の段階移行（expand/contract）とし、
  変更管理表に記録。廃止SQL（`drop ...`）は**戻し用SQLと事前バックアップ**をセットで用意してから流す。

### 1-4. 追加カラムの NULL は「作らない」か「必ず受ける」かに寄せる
- **理由**: SQL は `NULL` を伝播させる（`NULL - 3 = NULL`、`NULL + 5 = NULL`）。
  計算・集計に使う列に NULL が混じると、結果が黙って壊れる。
- **ルール**:
  - 計算・集計に使う数値列は **`not null default 0`** を基本にして NULL を作らない。
  - NULL を許容する列を計算に使うときは **`coalesce(col, 0)`** で必ず受ける。

### 1-5. 書き込み・制約は厳格、読み取りは寛容、と意識する
- 読み取り（SELECT）は列が増えても壊れにくい。壊れるのは主に **INSERT/UPDATE と制約（NOT NULL / FK / unique）**。
  スキーマ変更時は「書き込み側が通るか」を先に確認する。

## 2. TypeScript / アプリ実装

- **strict を維持**する。`any` は原則禁止。型は `src/lib/types.ts` に集約し、DBスキーマと対応させる。
- DBの列を追加したら、対応する **TS型も更新**する（型と実データのズレを残さない）。
- 入力値は Server Action 側で検証してから DB に渡す（クライアントの値を無条件に信用しない）。
- 秘密情報（`service_role` キー等）はクライアントに出さない。公開してよいのは `NEXT_PUBLIC_*`（anon key まで）。

---

## 3. Git / 変更管理

- 破壊的操作（`reset --hard` / force push / `drop`）の前に、戻す手段（バックアップ・戻しSQL）を用意する。
- スキーマや仕様に影響する変更は [変更管理表](./変更管理表.md) に記録。不具合は [バグ管理表](./バグ管理表.md) に残す。
- コミットは意味のある単位で。メッセージは「何を・なぜ」が分かるように書く。

---

## 4. このルールの位置づけ

特別なことは書いていない。**一般的なシステム開発の定石**を、本プロジェクトの構成（Next.js + Supabase）に
即して明文化しただけ。新しい機能（例: 在庫管理）を追加するときも、まずこの規約に沿って設計する。
